## What it does

A stack-agnostic planning and implementation framework that scales itself to the size of the change. Small fix? Quick mode, no ceremony. Multi-component feature? Full pipeline with requirement IDs, design docs, atomic task lists, and verification at each step.

The skill lives at `.workflow/skills/tlc-spec-driven/`. Source of truth is its `SKILL.md` and the references under `references/`.

<Note>
  This skill was authored by Felipe Rodrigues ([github.com/felipfr](https://github.com/felipfr)) and is reusable across any codebase, not just FluxPrompt. The team uses it for non-trivial feature work that benefits from a written spec and atomic commits.
</Note>

## The four phases

```
┌──────────┐   ┌──────────┐   ┌─────────┐   ┌─────────┐
│ SPECIFY  │ → │  DESIGN  │ → │  TASKS  │ → │ EXECUTE │
└──────────┘   └──────────┘   └─────────┘   └─────────┘
   required      optional*      optional*     required

* Auto-skipped when scope doesn't need it.
```

## Auto-sizing

The skill picks depth based on the change's scope. You don't have to declare anything up front — the agent reads the request and applies only what's needed.

| Scope | What | Specify | Design | Tasks | Execute |
| --- | --- | --- | --- | --- | --- |
| **Small** | ≤3 files, one sentence | Quick mode (skip pipeline) | — | — | — |
| **Medium** | Clear feature, <10 tasks | Spec (brief) | Skip — inline | Skip — implicit | Implement + verify |
| **Large** | Multi-component | Full spec + requirement IDs | Architecture + components | Full breakdown | Implement + verify per task |
| **Complex** | Ambiguity, new domain | Full spec + discuss gray areas | Research + architecture | Breakdown + parallel plan | Implement + interactive UAT |

**Rules of thumb:**

- **Specify and Execute are always required** — you always need to know WHAT and DO it.
- **Design is skipped** when the change is straightforward.
- **Tasks is skipped** when there are ≤3 obvious steps.
- **Discuss is triggered within Specify** when the agent finds ambiguous gray areas.
- **Interactive UAT is triggered within Execute** for user-facing features with complex behavior.

## When to use it

- Starting a new project (initializes `PROJECT.md` and `ROADMAP.md`)
- Mapping an existing codebase (writes 7 brownfield docs — stack, architecture, conventions, structure, testing, integrations, concerns)
- Planning a non-trivial feature
- Implementing with verification and atomic commits
- Tracking decisions, blockers, and deferred ideas across sessions (`STATE.md`)
- Pausing and resuming work (`references/session-handoff.md`)

## When **not** to use it

- Architecture decomposition analysis (use dedicated architecture skills)
- Pure technical design documents (use `create-technical-design-doc`)
- Single-line typos or one-file tweaks — the overhead exceeds the value

## Sub-agent delegation

The skill leans on sub-agents to keep the main context window lean.

| Activity | Delegate? | Why |
| --- | --- | --- |
| Research (design, brownfield mapping) | Yes | Large output, only the summary matters in main context |
| Implementing a task | Yes | File reads and test output consume context |
| Parallel `[P]` tasks | Yes (one per task) | The only way to actually run them in parallel |
| Sequential tasks with no `[P]` | Yes | Keeps implementation artifacts out of main context |
| Planning, task creation, validation reports | No | Needs full accumulated context |
| Quick-mode tasks | No | Too small to justify overhead |

## Knowledge verification chain

When researching or making technical decisions, the skill enforces a strict order:

1. **Codebase** — check existing code, conventions, patterns
2. **Project docs** — README, `docs/`, inline comments, `.specs/codebase/`
3. **Context7 MCP** — resolve library ID, then query for current API
4. **Web search** — official docs and reputable sources
5. **Flag as uncertain** — "I'm not certain about X — here's my reasoning, but verify"

<Warning>
  Step 5 is always flagged as uncertain. The skill explicitly forbids inventing APIs or patterns. Uncertainty is preferable to fabrication.
</Warning>
