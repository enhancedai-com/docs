## What this section is

This section documents the Claude Code tooling the engineering team uses every day on FluxPrompt: programmatic access to dev, staging, and production databases through an SSH bastion; custom subagents that diagnose incidents and create pull requests; and the playbooks that tie everything together.

It exists so that any teammate — not just the people who built the tooling — can ramp up, contribute, and reuse the same workflow without re-deriving how it works.

<Note>
  The tooling itself lives at the **workspace root** in `.workflow/` and `.claude/agents/`. This section explains how to use it; the source of truth for behavior is always the files in those folders.
</Note>

## Who this is for

- **New engineers** joining the team who need to set up their environment and start using the tooling — start with [Onboarding](/ai-workflows/internal-developer-workflows/onboarding).
- **Engineers already on the team** who want a reference for a specific skill, agent, or playbook — jump to the relevant card below.
- **People curious about how we use Claude Code** — read [Best practices](/ai-workflows/internal-developer-workflows/best-practices).

## Prerequisites

Before any of this works, you need:

- **Claude Code installed** and authenticated. See the [Claude Code docs](https://docs.claude.com/en/docs/claude-code/overview) if you have never used it.
- **The workspace cloned locally** with `back/`, `front/`, `docs/`, and the team-shared `.workflow/` folder.
- **Access to the AWS bastion authorized by an admin** — covered in [Onboarding](/ai-workflows/internal-developer-workflows/onboarding).
- **Node 18+, pnpm, and git** on your machine. AWS CLI v2 is optional but recommended for CloudWatch lookups.

## Three building blocks: Skill, Agent, Playbook

The tooling is layered. Knowing which layer you are in tells you where to look for problems and how to compose pieces together.

<Columns cols={3}>
  <Card title="Skill" icon="screwdriver-wrench">
    A reusable capability the model invokes via the `Skill` tool. Self-contained: instructions in `SKILL.md`, code in `templates/`, references in `references/`. Examples: `ec2-tunnel`, `ec2-tunnel-prod`, `db-migration`.
  </Card>
  <Card title="Agent" icon="user-gear">
    A specialized subagent defined in `.claude/agents/<name>.md`. Composes skills, tools, and rules to solve a specific class of problem. Examples: `db-investigator`, `db-setup`, `pr-creator`.
  </Card>
  <Card title="Playbook" icon="book-open">
    A documented diagnostic recipe a human follows (with or without Claude). Combines skills and agents in a known sequence for a known class of incident. Examples: Flux error investigation, memory leak diagnosis.
  </Card>
</Columns>

### When to reach for which

| Situation | Use |
| --- | --- |
| You want Claude to do a single discrete task (run a query, manage SSH keys) | A **skill** |
| You want Claude to drive a multi-step problem with judgement (diagnose, decide, reply) | An **agent** |
| You want to follow the same investigation a human did before | A **playbook** |
| You are doing something brand new | Compose skills directly in the main conversation |

## How files are organized

```
<workspace>/
├─ .workflow/                       Skills, KBs, rules. Shared with the team.
│  ├─ .env                          Your local credentials. NEVER committed.
│  ├─ .env.example                  Template — safe to share.
│  ├─ skills/
│  │  ├─ ec2-tunnel/                Dev/staging DB access via SSH tunnel
│  │  ├─ ec2-tunnel-prod/           Read-only prod DB access (triple-layer safety)
│  │  ├─ db-migration/              Data migration between environments
│  │  └─ tlc-spec-driven/           Adaptive feature planning
│  ├─ rules/                        Always-applied context for code work
│  └─ kb/                           Knowledge base (node specs, conventions)
└─ .claude/
   └─ agents/                       Custom subagents
      ├─ db-investigator.md         Diagnose "data missing" reports
      ├─ db-setup.md                Onboard new teammates
      └─ pr-creator.md              Create PRs from ClickUp tasks
```

<Warning>
  `.workflow/.env` is the only file that holds real secrets (SSH passphrase, DB passwords, bastion hostnames). It is git-ignored and must stay that way. If you accidentally stage it, abort the commit.
</Warning>
