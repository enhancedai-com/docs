## Dois métodos de autenticação

O servidor MCP de entrada do FluxPrompt suporta **dois** modos de autenticação. Qual deles você usa depende do seu cliente:

| Método | Quando usar | O que você faz |
|---|---|---|
| **OAuth 2.0** | Claude Desktop e Claude.ai (**obrigatório** — eles rejeitam tokens colados) e, opcionalmente, Claude Code / CLI e Cursor | Adiciona a URL do servidor como conector e faz login por uma tela de consentimento do FluxPrompt — **sem token** |
| **Service token estático** | Serviços de backend / máquina-a-máquina, a CLI, Cursor e n8n — clientes que não podem ou não devem executar um fluxo de consentimento no navegador | Gera um token e o coloca no header `Authorization` |

<Info>
**Usuários do Claude Desktop e do Claude.ai NÃO usam token.** Esses clientes rejeitam tokens bearer colados pelo usuário, então autenticam exclusivamente via OAuth. Se algum guia mandar você colar um token no Claude Desktop, ele está desatualizado — veja [Conectar Claude Desktop](/mcp/pt-br/connect/claude-desktop).
</Info>

## Método 1 — OAuth 2.0

OAuth é a escolha certa para qualquer cliente que consegue abrir um navegador. Você nunca vê nem manipula um token.

1. No seu cliente, adicione `https://mcp.fluxprompt.ai/mcp` como conector personalizado.
2. O cliente abre uma tela de login + consentimento do FluxPrompt.
3. Você faz login, **seleciona qual workspace** quer liberar e aprova.
4. O cliente armazena a autorização resultante automaticamente e a renova quando necessário.

Isso é **obrigatório** para Claude Desktop e Claude.ai, e **suportado** por Claude Code / CLI e Cursor. Veja os guias por cliente para os passos exatos.

## Método 2 — Service token estático

Para clientes máquina-a-máquina (serviços de backend), a CLI, o Cursor e o n8n, você usa um **service token** em vez de um fluxo no navegador.

### Gerar um service token

O dono do workspace gera um na interface do FluxPrompt:

1. Abra **Settings → MCP Tokens** — ou vá direto adicionando `?settings=mcpTokens` à URL do seu app FluxPrompt (ex.: `https://app.fluxprompt.ai/dashboard-beta?settings=mcpTokens`).
2. Crie um novo token e escolha o workspace a que ele dá acesso.
3. Copie o token — ele começa com `mcp_` e é **exibido apenas uma vez**. Guarde-o em local seguro.

### Usar o token

O token vai no header `Authorization` de cada requisição MCP:

```
Authorization: Bearer mcp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

Normalmente você não digita esse header à mão — seu cliente o lê de um arquivo de configuração (veja os guias de conexão para [Cursor](/mcp/pt-br/connect/cursor), [n8n](/mcp/pt-br/connect/n8n) e [desenvolvimento local](/mcp/pt-br/connect/local-dev)).

## Segurança do service token

<Warning>
Um service token concede acesso aos agentes do seu workspace nos escopos com que foi gerado (leitura e, opcionalmente, escrita e execução). Trate-o como uma senha:
- Nunca o commite em um repositório Git.
- Nunca o cole em um documento público ou screenshot.
- Nunca o compartilhe no Slack, e-mail ou em qualquer mensagem que outras pessoas possam ver.
</Warning>

Se você acredita que um service token foi exposto:

1. Revogue-o em **Settings → MCP Tokens**.
2. Gere um novo.
3. Atualize seus arquivos de configuração do cliente com o novo token.

## Escopos

Um token (ou sessão OAuth) carrega um ou mais **escopos** que decidem quais tools ele pode chamar. São três:

| Escopo | Concede | Tools de exemplo |
|---|---|---|
| `read` | Descobrir e inspecionar — nunca altera nada | `list_agents`, `get_agent`, `list_node_types`, `get_node_type`, `list_folders`, `list_versions` |
| `write` | Criar e editar agentes, pastas, nodes, conexões e versões | `add_node`, `update_node`, `connect_nodes`, `create_agent`, `create_version`, `publish_version` |
| `execute` | Executar agentes e gerenciar execuções | `run_agent`, `get_run_status`, `cancel_run` |

`write` e `execute` se apoiam em `read` — um token precisa de `read` para descobrir os agentes e tipos de node sobre os quais vai agir.

<Info>
Os escopos são necessários, mas não suficientes. As tools de `write` e `execute` também exigem as **feature flags de workspace** correspondentes (`mcp.writes.enabled`, `mcp.execute.enabled`) ligadas. Um token com escopo `write` ainda não consegue alterar um workspace cuja flag `mcp.writes.enabled` esteja desligada. Veja [O que a IA pode fazer](/mcp/pt-br/overview#o-que-a-ia-pode-fazer) para o modelo de dry-run e flags.
</Info>

Independentemente do escopo, o servidor MCP **não** acessa faturamento, usuários ou configurações do workspace — ele enxerga apenas o que o usuário autenticado vê na interface do FluxPrompt.

## Perguntas frequentes

<AccordionGroup>
  <Accordion title="Qual método devo usar?">
    Se o seu cliente é Claude Desktop ou Claude.ai, você precisa usar OAuth. Para serviços de backend, a CLI ou o n8n, use um service token. Cursor e Claude Code podem usar qualquer um dos dois — OAuth é o mais simples se você consegue abrir um navegador.
  </Accordion>
  <Accordion title="Posso ter múltiplos service tokens para o mesmo workspace?">
    Sim — cada token é independente. Você pode ter um token por cliente (ex.: um para n8n, outro para um serviço de backend) e revogá-los individualmente em **Settings → MCP Tokens**.
  </Accordion>
  <Accordion title="O service token expira?">
    Service tokens não expiram automaticamente. Eles permanecem válidos até serem revogados. Recomendamos revogar tokens que você não usa mais.
  </Accordion>
  <Accordion title="Posso usar minha API key comum do FluxPrompt no lugar?">
    Não. O servidor MCP usa um sistema de autenticação separado. Sua API key comum não funcionará.
  </Accordion>
  <Accordion title="O que o servidor MCP enxerga do meu workspace?">
    Ele vê os mesmos agentes e estrutura que o usuário autenticado consegue ver dentro da interface do FluxPrompt. Ele não tem acesso a credenciais, faturamento ou gerenciamento de usuários.
  </Accordion>
</AccordionGroup>
