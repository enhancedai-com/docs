## O que é o FluxPrompt MCP?

**FluxPrompt MCP** é um servidor [Model Context Protocol](https://modelcontextprotocol.io) de entrada. Ele permite que clientes de IA externos — Claude Desktop, Cursor, n8n e outros — se conectem ao seu workspace FluxPrompt e interajam com seus agentes por linguagem natural.

Em vez de navegar pela interface do FluxPrompt, seu cliente de IA pode fazer perguntas como:

- *"Liste todos os agentes do meu workspace"*
- *"Mostre os nodes dentro do agente Invoice Parser"*
- *"Quais tipos de node o FluxPrompt suporta?"*

## Autenticação em resumo

O FluxPrompt MCP suporta **dois** métodos de autenticação, e o correto depende do seu cliente:

- **OAuth 2.0** — você adiciona a URL do servidor como conector, faz login por uma tela de consentimento do FluxPrompt e escolhe um workspace. Nenhum token é colado. É **obrigatório** para Claude Desktop e Claude.ai (que rejeitam tokens colados) e também é suportado por Claude Code / CLI e Cursor.
- **Service token estático** — o dono do workspace gera um token (começa com `mcp_`) em **Settings → MCP Tokens** e o coloca no header `Authorization`. É para clientes máquina-a-máquina, a CLI, o Cursor e o n8n — clientes que não podem ou não devem executar um fluxo de consentimento no navegador.

| Cliente | OAuth | Service token estático |
|---------|-------|------------------------|
| Claude Desktop | ✅ obrigatório | ❌ não suportado |
| Claude.ai | ✅ obrigatório | ❌ não suportado |
| Claude Code / CLI | ✅ | ✅ |
| Cursor | ✅ | ✅ |
| n8n | parcial | ✅ recomendado |
| Serviços de backend (M2M) | ❌ (sem navegador) | ✅ |

Veja [Autenticação](/mcp/pt-br/authentication) para todos os detalhes.

## O que a IA pode fazer

O FluxPrompt MCP expõe **23 tools** mais os primitivos **Prompts** e **Resources** da spec MCP (revisão 2025-11-25). As tools cobrem todo o ciclo de vida de um agente: descobrir os blocos de construção, montar e editar flows, tirar snapshots e publicar versões, e executar agentes.

### Referência de tools

As tools são agrupadas pelo [escopo](/mcp/pt-br/authentication#escopos) que um token precisa para chamá-las.

| Escopo | Tools |
|---|---|
| `read` | `list_agents`, `get_agent`, `list_node_types`, `get_node_type`, `list_folders`, `list_versions` |
| `write` | `create_agent`, `remove_agent`, `update_agent`, `move_agent`, `add_node`, `update_node`, `remove_node`, `connect_nodes`, `disconnect_nodes`, `create_folder`, `rename_folder`, `remove_folder`, `create_version`, `publish_version` |
| `execute` | `run_agent`, `get_run_status`, `cancel_run` |

<Note>
As tools de `read` estão sempre disponíveis. As tools de `write` e `execute` também exigem a feature flag de workspace correspondente (veja [Mutações são dry-run por padrão](#mutações-são-dry-run-por-padrão)).
</Note>

### Como a IA deve consumir tipos de node

A descoberta de tipos de node usa **divulgação progressiva** (progressive disclosure) — duas tools que trabalham juntas para que a IA carregue apenas o que precisa:

<Steps>
<Step title="Chame list_node_types primeiro">
  Retorna um **resumo leve** de cada tipo de node: `type`, `label`, `category` e `description`. Sem schemas. É barato o suficiente para carregar o catálogo inteiro no contexto.
</Step>
<Step title="Chame get_node_type só para os tipos que vai usar">
  Passe um único `type` e receba de volta o `inputSchema` e o `outputSchema` completos daquele tipo de node. Chame uma vez por tipo de node que pretende adicionar ou configurar.
</Step>
</Steps>

Isso mantém o uso de tokens baixo: a IA varre um catálogo compacto e busca os schemas detalhados apenas para os poucos tipos de node que a tarefa envolve — em vez de puxar todos os schemas de antemão. Veja [`list_node_types`](/mcp/pt-br/tools/list-node-types).

### Mutações são dry-run por padrão

Toda tool de `write` roda como **dry-run** a menos que você opte por aplicar. Um dry-run valida a mudança e retorna o que *aconteceria* sem persistir nada.

Para de fato aplicar uma mutação, passe ambos:

- `dryRun: false`
- `idempotencyKey` — uma string fornecida pelo chamador. Repetir a mesma chave retorna o resultado original em vez de aplicar a mudança duas vezes, então uma chamada repetida ou duplicada é segura.

As mutações também são limitadas por feature flags de workspace e escopos de token:

| Limite | Controla |
|---|---|
| `mcp.writes.enabled` (flag) | Se as tools de `write` podem rodar neste workspace |
| `mcp.execute.enabled` (flag) | Se as tools de `execute` podem rodar neste workspace |
| `write` / `execute` (escopo do token) | Se *este* token pode chamar essas tools |

Tanto a flag quanto o escopo precisam permitir a chamada. Um dry-run bem-sucedido confirma que a mudança é válida; a aplicação ainda falha se a flag ou o escopo estiverem ausentes.

### Versionamento

Agentes têm uma **cópia de trabalho** ao vivo e um histórico de **versões** (snapshots). Três tools as gerenciam:

| Tool | Escopo | O que faz |
|---|---|---|
| `list_versions` | `read` | Lista o histórico de versões de um agente. |
| `create_version` | `write` | Tira um snapshot da cópia de trabalho atual como uma nova versão. |
| `publish_version` | `write` | Marca uma versão como a publicada (deployed). |

As tools de `write` aceitam um `versionId` opcional. Quando informado, a edição mira o histórico daquela versão em vez da cópia de trabalho ao vivo — assim a IA pode revisar um snapshot sem tocar no que está ao vivo.

### Primitivos MCP: quem controla o quê

O FluxPrompt MCP expõe os três primitivos do MCP. Eles diferem em **qual ator os invoca**, o que importa para como a IA os usa:

| Primitivo | Controlado por | Métodos | O que o FluxPrompt expõe |
|---|---|---|---|
| **Tools** | O modelo (a IA decide) | `tools/list`, `tools/call` | As 23 tools acima — descoberta e ações. É assim que a IA consome tipos de node de forma autônoma (`list_node_types` → `get_node_type`). |
| **Prompts** | O usuário | `prompts/list`, `prompts/get` | Receitas guiadas que um usuário escolhe: `build_flow`, `edit_agent`, `organize_workspace`, `run_agent_recipe`. |
| **Resources** | A aplicação / o usuário | `resources/list`, `resources/read` | Contexto que um usuário ou app anexa: `fluxprompt://models` (modelos de IA disponíveis) e `fluxprompt://node-types` (o catálogo completo de tipos de node). |

<Warning>
Os Resources **não** são invocados automaticamente pelo modelo. O resource `fluxprompt://node-types` serve para um usuário ou app anexar o catálogo como contexto — não é o caminho de descoberta autônoma. Para a IA descobrir tipos de node por conta própria, ela usa as **tools** (`list_node_types` e depois `get_node_type`), não o resource.
</Warning>

## Como funciona

```
Seu cliente de IA (Claude Desktop / Cursor / n8n)
        │
        │  JSON-RPC sobre HTTPS
        │  Sessão OAuth  ou  Authorization: Bearer <service token>
        ▼
Servidor FluxPrompt MCP  (mcp.fluxprompt.ai)
        │
        │  lê seu workspace
        ▼
FluxPrompt API  ──▶  Seus agentes, nodes e flows
```

1. Você se autentica via **OAuth** (consentimento no navegador) ou um **service token com escopo de workspace**, dependendo do seu cliente.
2. Seu cliente de IA se conecta a `https://mcp.fluxprompt.ai/mcp`.
3. Ele chama as tools — para descobrir, montar, versionar ou executar agentes — e recebe respostas estruturadas.
4. Cada chamada é registrada no log de auditoria do FluxPrompt.

## Glossário rápido

| Termo | Significado |
|-------|-------------|
| **MCP** | Model Context Protocol — um padrão aberto para conectar clientes de IA a ferramentas e dados externos. |
| **OAuth** | Um fluxo de consentimento no navegador em que você faz login no FluxPrompt e concede a um cliente acesso a um workspace. Nenhum token é colado. Obrigatório para Claude Desktop e Claude.ai. |
| **Service token** | Uma string secreta com escopo limitado ao seu workspace (começa com `mcp_`), gerada em **Settings → MCP Tokens**. Enviada no header `Authorization`. Usada por clientes máquina-a-máquina, a CLI e o n8n. Diferente da sua API key do FluxPrompt. |
| **Tool** | Uma ação nomeada que o servidor MCP expõe (ex.: `list_agents`). Seu cliente de IA chama as tools em seu nome. |
| **Workspace** | O contexto da sua empresa ou equipe no FluxPrompt. Um token MCP = um workspace. |
| **Streamable HTTP** | O protocolo de transporte que o FluxPrompt MCP utiliza. Um único endpoint HTTPS que aceita requisições JSON-RPC 2.0. |

## Antes de começar

Você vai precisar de:

- Uma **conta FluxPrompt** com pelo menos um agente no seu workspace.
- Um cliente compatível com OAuth (Claude Desktop, Claude.ai, Claude Code, Cursor) ou um **service token** de **Settings → MCP Tokens** para clientes baseados em token (n8n, serviços de backend).
- Um dos clientes de IA suportados instalado.

<Note>
Este guia assume que o servidor FluxPrompt MCP está em `https://mcp.fluxprompt.ai/mcp`. Se a sua equipe usa um endpoint dedicado, utilize esse no lugar.
</Note>
