## Two authentication methods

FluxPrompt's inbound MCP server supports **two** ways to authenticate. Which one you use depends on your client:

| Method | When to use it | What you do |
|---|---|---|
| **OAuth 2.0** | Claude Desktop and Claude.ai (**required** ‚Äî they reject pasted tokens), and optionally Claude Code / CLI and Cursor | Add the server URL as a connector and log in through a FluxPrompt consent screen ‚Äî **no token** |
| **Static service token** | Backend / machine-to-machine services, the CLI, Cursor, and n8n ‚Äî clients that can't or shouldn't run a browser consent flow | Generate a token and put it in the `Authorization` header |

<Info>
**Claude Desktop and Claude.ai users do NOT use a token.** Those clients reject user-pasted bearer tokens, so they authenticate exclusively via OAuth. If a guide tells you to paste a token into Claude Desktop, it is out of date ‚Äî see [Connect Claude Desktop](/mcp/connect/claude-desktop).
</Info>

## Method 1 ‚Äî OAuth 2.0

OAuth is the right choice for any client that can open a browser. You never see or handle a token.

1. In your client, add `https://mcp.fluxprompt.ai/mcp` as a custom connector.
2. The client opens a FluxPrompt login + consent screen.
3. You log in, **select which workspace** to grant access to, and approve.
4. The client stores the resulting authorization automatically and refreshes it as needed.

This is **required** for Claude Desktop and Claude.ai, and **supported** by Claude Code / CLI and Cursor. See the per-client guides for the exact steps.

## Method 2 ‚Äî Static service token

For machine-to-machine clients (backend services), the CLI, Cursor, and n8n, you use a **service token** instead of a browser flow.

### Generate a service token

The workspace owner generates one in the FluxPrompt UI:

1. Open **Settings ‚Üí MCP Tokens** ‚Äî or jump straight there by appending `?settings=mcpTokens` to your FluxPrompt app URL (e.g. `https://app.fluxprompt.ai/dashboard-beta?settings=mcpTokens`).
2. Create a new token and pick the workspace it grants access to.
3. Copy the token ‚Äî it starts with `mcp_` and is **shown only once**. Store it somewhere safe.

### Use the token

The token goes in the `Authorization` header of every MCP request:

```
Authorization: Bearer mcp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

You usually don't type this header by hand ‚Äî your client reads it from its config file (see the connection guides for [Cursor](/mcp/connect/cursor), [n8n](/mcp/connect/n8n), and [local development](/mcp/connect/local-dev)).

## Service token security

<Warning>
A service token gives access to your workspace agents at the scopes it was granted (read, and optionally write and execute). Treat it like a password:
- Never commit it to a Git repository.
- Never paste it in a public document or screenshot.
- Never share it on Slack, email, or in any message that others can see.
</Warning>

If you believe a service token has been exposed:

1. Revoke it from **Settings ‚Üí MCP Tokens**.
2. Generate a new one.
3. Update your client config files with the new token.

## Scopes

A token (or OAuth session) carries one or more **scopes** that decide which tools it can call. There are three:

| Scope | Grants | Example tools |
|---|---|---|
| `read` | Discover and inspect ‚Äî never changes anything | `list_agents`, `get_agent`, `list_node_types`, `get_node_type`, `list_folders`, `list_versions` |
| `write` | Create and edit agents, folders, nodes, connections, and versions | `add_node`, `update_node`, `connect_nodes`, `create_agent`, `create_version`, `publish_version` |
| `execute` | Run agents and manage runs | `run_agent`, `get_run_status`, `cancel_run` |

`write` and `execute` build on `read` ‚Äî a token needs `read` to discover the agents and node types it will act on.

<Info>
Scopes are necessary but not sufficient. `write` and `execute` tools also require the matching **workspace feature flags** (`mcp.writes.enabled`, `mcp.execute.enabled`) to be turned on. A token with the `write` scope still can't mutate a workspace whose `mcp.writes.enabled` flag is off. See [What the AI can do](/mcp/overview#what-the-ai-can-do) for the dry-run and flag model.
</Info>

Regardless of scope, the MCP server **cannot** access billing, users, or workspace settings ‚Äî it sees only what the authenticated user sees in the FluxPrompt UI.

## FAQ

<AccordionGroup>
  <Accordion title="Which method should I use?">
    If your client is Claude Desktop or Claude.ai, you must use OAuth. For backend services, the CLI, or n8n, use a service token. Cursor and Claude Code can use either ‚Äî OAuth is simplest if you can open a browser.
  </Accordion>
  <Accordion title="Can I have multiple service tokens for the same workspace?">
    Yes ‚Äî each token is independent. You can have one token per client (e.g. one for n8n, one for a backend service) and revoke them individually from **Settings ‚Üí MCP Tokens**.
  </Accordion>
  <Accordion title="Does a service token expire?">
    Service tokens do not expire automatically. They remain valid until revoked. We recommend revoking tokens you no longer use.
  </Accordion>
  <Accordion title="Can I use my regular FluxPrompt API key instead?">
    No. The MCP server uses a separate authentication system. Your regular API key will not work.
  </Accordion>
  <Accordion title="What does the MCP server see about my workspace?">
    It sees the same agents and structure that the authenticated user can see inside the FluxPrompt UI. It does not have access to credentials, billing, or user management.
  </Accordion>
</AccordionGroup>
