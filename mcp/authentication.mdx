## Two authentication methods

FluxPrompt's inbound MCP server supports **two** ways to authenticate. Which one you use depends on your client:

| Method | When to use it | What you do |
|---|---|---|
| **OAuth 2.0** | Claude Desktop and Claude.ai (**required** — they reject pasted tokens), and optionally Claude Code / CLI and Cursor | Add the server URL as a connector and log in through a FluxPrompt consent screen — **no token** |
| **Static service token** | Backend / machine-to-machine services, the CLI, Cursor, and n8n — clients that can't or shouldn't run a browser consent flow | Generate a token and put it in the `Authorization` header |

<Info>
**Claude Desktop and Claude.ai users do NOT use a token.** Those clients reject user-pasted bearer tokens, so they authenticate exclusively via OAuth. If a guide tells you to paste a token into Claude Desktop, it is out of date — see [Connect Claude Desktop](/mcp/connect/claude-desktop).
</Info>

## Method 1 — OAuth 2.0

OAuth is the right choice for any client that can open a browser. You never see or handle a token.

1. In your client, add `https://mcp.fluxprompt.ai/mcp` as a custom connector.
2. The client opens a FluxPrompt login + consent screen.
3. You log in, **select which workspace** to grant access to, and approve.
4. The client stores the resulting authorization automatically and refreshes it as needed.

This is **required** for Claude Desktop and Claude.ai, and **supported** by Claude Code / CLI and Cursor. See the per-client guides for the exact steps.

## Method 2 — Static service token

For machine-to-machine clients (backend services), the CLI, Cursor, and n8n, you use a **service token** instead of a browser flow.

### Generate a service token

The workspace owner generates one in the FluxPrompt UI:

1. Open [**Settings → MCP Tokens**](https://app.fluxprompt.ai/dashboard-beta?settings=mcpTokens).
2. Create a new token and pick the workspace it grants access to.
3. Copy the token — it starts with `mcp_` and is **shown only once**. Store it somewhere safe.

### Use the token

The token goes in the `Authorization` header of every MCP request:

```
Authorization: Bearer mcp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

You usually don't type this header by hand — your client reads it from its config file (see the connection guides for [Cursor](/mcp/connect/cursor), [n8n](/mcp/connect/n8n), and [local development](/mcp/connect/local-dev)).

## Service token security

<Warning>
A service token gives full read access to your workspace agents. Treat it like a password:
- Never commit it to a Git repository.
- Never paste it in a public document or screenshot.
- Never share it on Slack, email, or in any message that others can see.
</Warning>

If you believe a service token has been exposed:

1. Revoke it from [**Settings → MCP Tokens**](https://app.fluxprompt.ai/dashboard-beta?settings=mcpTokens).
2. Generate a new one.
3. Update your client config files with the new token.

## Scopes (Phase 1)

Both authentication methods currently grant **read-only** access. They can:

- ✅ List agents in your workspace
- ✅ Fetch the full structure of any agent
- ✅ List all available node types

They cannot:

- ❌ Create or edit agents
- ❌ Run agents or start executions
- ❌ Access billing, users, or workspace settings

Write access (`add_node`, `update_node`, `remove_node`) will be available in Phase 2 with a separate scope.

## FAQ

<AccordionGroup>
  <Accordion title="Which method should I use?">
    If your client is Claude Desktop or Claude.ai, you must use OAuth. For backend services, the CLI, or n8n, use a service token. Cursor and Claude Code can use either — OAuth is simplest if you can open a browser.
  </Accordion>
  <Accordion title="Can I have multiple service tokens for the same workspace?">
    Yes — each token is independent. You can have one token per client (e.g. one for n8n, one for a backend service) and revoke them individually from [**Settings → MCP Tokens**](https://app.fluxprompt.ai/dashboard-beta?settings=mcpTokens).
  </Accordion>
  <Accordion title="Does a service token expire?">
    Service tokens do not expire automatically. They remain valid until revoked. We recommend revoking tokens you no longer use.
  </Accordion>
  <Accordion title="Can I use my regular FluxPrompt API key instead?">
    No. The MCP server uses a separate authentication system. Your regular API key will not work.
  </Accordion>
  <Accordion title="What does the MCP server see about my workspace?">
    It sees the same agents and structure that the authenticated user can see inside the FluxPrompt UI. It does not have access to credentials, billing, or user management.
  </Accordion>
</AccordionGroup>
