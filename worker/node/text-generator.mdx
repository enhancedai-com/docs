## Overview

The **text generator** node produces LLM output using the Vercel AI SDK (`streamText`, consumed internally). The worker loads node configuration from Postgres, resolves the model via `system_ai_models`, builds a unified system prompt and messages, calls the provider, persists results on the node row, and returns token usage to the workflow for billing.

[`textGeneratorActivity`](https://github.com/enhancedai-com/Flux-Prompt-Worker/blob/main/src/modules/temporal/activities/activities.service.ts) delegates to [`TextGeneratorService.process`](https://github.com/enhancedai-com/Flux-Prompt-Worker/blob/main/src/modules/nodes/text-generator/text-generator.service.ts).

For **product and UI** details (inputs, provider knobs, model capability tables), see [AI Text Generator](/worker/text-generator).

## When it runs

`textGeneratorActivity` is invoked from the [`processSingleNode`](/worker/workflows/process-single-node) workflow when the node type is **text generator** (`NodeType.TEXT_GENERATOR`). The workflow typically calls [`validateModelAccessActivity`](/worker/activities/validate-model-access) before execution and [`chargeTokensActivity`](/worker/activities/charge-tokens) afterward using the returned token counts.

## Activity signature

```typescript
async textGeneratorActivity({
  nodeId: string,
  sessionId: string,
  userId: string,
}): Promise<TextGeneratorResponse>
```

`TextGeneratorResponse`:

```typescript
{
  output?: string;
  inputTokens?: number;
  outputTokens?: number;
}
```

The activity returns only these fields; richer state (generated text, logs, previews, memory) is written to `flows_nodes.data` inside `TextGeneratorService.process`.

## Module structure

The text generator module is split into focused pure functions, each in its own file:

| File | Responsibility |
| ---- | -------------- |
| `text-generator.service.ts` | Orchestrator: fetch node, build payload, call `streamText`, persist output |
| `build-system-prompt.ts` | Unified system prompt from persona, instructions, context, and prompt |
| `build-messages.ts` | Messages array from filtered memory entries |
| `build-provider-options.ts` | Provider-specific options (temperature, topP, reasoning, tokens) |
| `build-web-search-tools.ts` | Web search tools per provider |
| `build-logs.ts` | Structured log type definition (logs are built inline during stream consumption) |
| `extract-multimodal-content.ts` | URL detection in prompt, content part classification |
| `process-memory-filter.ts` | Memory filtering by entry count or word count |
| `text-generator.types.ts` | TypeScript interfaces and enums |
| `text-generator.module.ts` | NestJS module definition |

All files except the service export pure functions with no NestJS dependency.

## End-to-end flow

1. **Load node** — `SELECT` from `flows_nodes` by `nodeId`; read `data` (prompt, provider fields, memory flags, etc.).
2. **Resolve model field** — From `data.AiSystem`, pick which property holds the model id (`aiModel`, `groqModel`, `geminiModel`, `anthropicModel`, or `perplexityType`).
3. **Load model metadata** — `SELECT` from `system_ai_models` where `modelName` matches (`aiCompany`, limits, `supportsReasoning`, etc.).
4. **Instantiate model** — Map `aiCompany` to a Vercel AI SDK client (`openai`, `anthropic`, `gemini`, `perplexity`, `xai`, `groq`) and return a `LanguageModel`. Anthropic ids are normalized via an internal map.
5. **Extract multimodal content** — If `multimodalEnabled`, extract file URLs from the prompt, classify as `image` or `file`, and return cleaned prompt without URLs.
6. **Build system prompt** — Compose a single string with four sections: `# Persona`, `# Instructions`, `# Context`, `# Task`. Each section resolves from `*Data` arrays and fallback string fields. If all sections are empty, a default generic prompt is used.
7. **Build messages** — When memory is enabled, filter the active session's memories and convert to `ModelMessage[]`. When messages are empty (no memory, no multimodal), a fallback user message is added to trigger the model.
8. **Build tools and options** — Attach web search tools if enabled. Build provider-specific options (temperature, max tokens, reasoning effort, stop sequences, etc.).
9. **Call LLM** — `streamText` with model, system, messages, tools, and options. The stream is consumed internally; the final text, token usage, and steps are awaited.
10. **Build logs** — Extract structured logs from the first step result: `reasoning`, `sources`, `toolCalls`, `toolResults`, `finishReason`. Saved as a JSON string.
11. **Persist** — `UPDATE flows_nodes` with `text`, `logs`, `executionStatus: "COMPLETED"`, accumulated `nodeUsedTokens`, appended `previewResponses`, and updated `userSessionsChat` when memory is enabled.
12. **Return** — `output`, `inputTokens`, `outputTokens` from `result.totalUsage`.

## System prompt structure

All stable configuration is placed in the system prompt. The `buildSystemPrompt` function composes these sections in order, skipping empty ones:

```text
# Persona
[from personaData array and/or persona string]

# Instructions
[from instructionData array and/or instruction string]

# Context
[from contentData array and/or content string]

# Task
[from promptData array and/or prompt string]
```

Each `*Data` array field takes priority, with the corresponding string field appended as a fallback. When multimodal URLs are extracted from the prompt, the cleaned prompt (without URLs) is used for the `# Task` section.

## Persisted node `data` highlights

| Field | Role |
| ----- | ----- |
| `text` | Model output string |
| `logs` | JSON string with structured execution data (see logs structure below) |
| `executionStatus` | Set to `COMPLETED` on success |
| `nodeUsedTokens` | Accumulated total token count (previous value + input + output tokens from this run) |
| `previewResponses` | Previous outputs plus the new output appended (last 3 kept) |
| `userSessionsChat` | When `memory` is true, the current session's `memories` gain new user/assistant entries |

## Logs structure

Logs are saved as a JSON string containing:

```typescript
{
  reasoning: string | null;    // thinking/reasoning text from the model
  sources: Array<{             // web search sources
    url: string;
    title: string;
    id: string;
  }>;
  toolCalls: Array<any>;       // tool calls made during generation
  toolResults: Array<any>;     // tool call results
  finishReason: string;        // "stop", "length", "tool-calls", etc.
}
```

Fields are captured during stream consumption from the following event types: `text-delta`, `reasoning-delta`, `source`, `tool-call`, `tool-result`, and `finish`.

## Web search

When `node.webSearch` is true, provider-specific tools are attached:

| `AiSystem` | Tool key | SDK call |
| ---------- | -------- | -------- |
| OpenAI | `web_search` | `openai.tools.webSearch({})` |
| Anthropic | `web_search` | `anthropic.tools.webSearch_20250305({})` |
| Gemini | `enterprise_web_search` | `google.tools.googleSearch({})` |

Other providers (Perplexity, xAI, Groq) return no tools when web search is requested.

## Multimodal

When `multimodalEnabled` is true, URLs in the prompt are extracted via regex, classified by extension, and passed as Vercel AI SDK content parts:

- **Images** (jpg, jpeg, png, gif, webp) → `{ type: "image", image: "<url>" }`
- **Other files** (pdf, audio, video, etc.) → `{ type: "file", data: "<url>", mediaType: "application/octet-stream" }`

Maximum **3 files** per execution. No base64 conversion or file size validation is performed; the SDK and provider handle content natively. URLs are removed from the prompt to avoid duplication in the system prompt.

Content parts are injected into the last user message. If no user messages exist, a new user message is created with the content parts.

## Session memory

When `memory` is true, `sessionId` identifies which session to load from `userSessionsChat`. The `processMemoryFilter` function applies one of five filter strategies:

| `selectedType` | Behavior |
| -------------- | -------- |
| `Load First X Entries` | First N memory entries (by `selectedIndex`) |
| `Load Last X Entries` | Last N memory entries (by `selectedIndex`) |
| `Load First X Words` | First N words across all memories (by `selectedNumber`) |
| `Load Last X Words` | Last N words across all memories (by `selectedNumber`) |
| `Load All Entries` | All memories with valid content (default) |

Filtered memories are converted to `ModelMessage[]` with their original roles. After generation, if a prompt was provided, new user and assistant memory entries are appended to the active session with timestamps.

## Provider-specific generation options

`buildProviderOptions` branches on `modelInfo.aiCompany` (lowercased):

- **Anthropic** — `anthropicToken`, `anthropicTemperature`, `anthropicTopP`, `anthropicTopK`. Temperature/topP/topK disabled when reasoning is active.
- **Gemini** — `geminiMaxOutputTokens`, `temperature`, `topP`, `topK`. Temperature/topP/topK disabled when reasoning is active.
- **OpenAI** — `aiTokens`, `aiTemperature`, `aiPp` (topP), `aiFp` (topK), `aiEffort` → `providerOptions.openai.reasoningEffort`.
- **Perplexity** — `max_tokens`/`perplexityTokens`, temperature, topP, topK, `search_context_size` in provider options.
- **xAI** — `aiTokens`, `aiTemperature`.
- **Groq** — `groqMaxTokens`, temperature/topP (with Gemini fallbacks), `seed`, `groqStop` as stop sequences.

**Reasoning logic:** When `modelInfo.supportsReasoning` is true and either `node.reasoning` is true or `modelInfo.reasoningAlwaysOn` is true, temperature/topP/topK are omitted and max output tokens default to the model maximum. The model output may include `reasoningText` separately from `text`; the final output selects `text` first, falling back to `reasoningText`.

**Auto token:** When `autoToken` is true, max output tokens are set to `modelInfo.maxOutputToken` regardless of the configured value.
