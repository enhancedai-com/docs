## What integrations are

Third-party integrations connect your flows to external services. You add a **Third Party Integration** node, pick a provider, and the node runs in one of two modes:

- **Pull** — read data from the service into your flow (for example, query a Notion database or list Slack channels).
- **Push** — send data from your flow to the service (for example, post a Slack message or create a Stripe invoice).

Some providers support both modes; others support only one.

## Available integrations

Integrations come in two families:

- **Third Party Integration node providers** use the shared adapter framework. You add a single **Third Party Integration** node and select the provider.
- **Dedicated-node integrations** (Google Workspace, Microsoft 365, HubSpot) predate that framework. They are accessed through their own flow nodes — **Pull Data**, **Push Data**, **HubSpot CRM**, and the email/output node — and connect through their own settings cards.

| Provider | Node | Connection | Pull | Push |
|---|---|---|---|---|
| WordPress | Third Party Integration | Credentials (site URL + Application Password) | Yes | Yes |
| Slack | Third Party Integration | OAuth | Yes | Yes |
| Notion | Third Party Integration | OAuth | Yes | Yes |
| Stripe | Third Party Integration | Secret API key | Yes | Yes |
| Zapier | Third Party Integration | Webhook URL | Yes | Yes |
| MCP | Third Party Integration | API key / token (per transport) | Yes (list tools) | No |
| ClickUp | Third Party Integration | OAuth | Yes | Yes |
| Google Workspace | Pull Data / Push Data / email | OAuth (Google account) | Yes | Yes |
| Microsoft 365 | Pull Data / Push Data / email | OAuth (Microsoft account) | Yes | Yes |
| HubSpot CRM | HubSpot CRM | OAuth (HubSpot account) | Yes | No (read-only node) |

<Note>
The frontend config labels some providers as `oauth2` for grouping, but WordPress and Stripe actually connect with entered credentials (no OAuth flow), and Zapier is webhook-based. Google, Microsoft 365, and HubSpot also appear as Third Party Integration entries, but those are non-functional stubs — the working integrations live in their dedicated nodes. Each provider page documents the real connection method and node.
</Note>

## Connection lifecycle

Every integration follows three phases:

<Steps>
  <Step title="Connect">
    You save credentials once — an OAuth token, an API key, or a webhook URL — to link your account. Each user has at most one connection per provider.
  </Step>
  <Step title="Use in a flow">
    You add the integration's node — the Third Party Integration node for adapter-based providers, or a dedicated Pull Data, Push Data, HubSpot CRM, or email node — select the provider, and choose what to read or write.
  </Step>
  <Step title="Share (optional)">
    Teammates can use your connection through the flow without seeing your raw credentials. The node records who connected it and resolves their credentials at run time.
  </Step>
</Steps>

### Connection states

| State | Meaning |
|---|---|
| Connected by me | You authenticated the integration yourself. |
| Connected by another user | A teammate connected and shared access through the flow. |
| Not connected | No credentials exist; the node cannot run until someone connects. |

## Credential storage

Connections are stored in the `user_integrations` table, with a unique constraint on `(userId, provider)`. OAuth tokens and entered credentials are encrypted at rest with AES-256-GCM and decrypted only at request time inside the provider adapter. For the full adapter pattern, credential resolution, and how to add a new provider, see the [third-party integrations architecture](/main-api/specific-features/third-party-integrations-architecture).

## Run modes in the node

| Action | Behavior |
|---|---|
| Get Data | Runs only the integration node and returns its output. Use it to preview results while building a flow. |
| Run flow | Runs the entire flow; the integration node executes in sequence with every other node. |
